# no-store 이미지 Service Worker 캐싱 데모

`Cache-Control: no-store` 헤더로 내려오는 이미지를 브라우저 Service Worker의 `fetch` 이벤트에서 가로채서 `Cache API`에 저장/재사용하는 예제입니다.

## 구성

- **backend**: Node.js + Express ([server](server))
- **frontend**: React + TypeScript + TailwindCSS + Vite ([client](client))

## 핵심 시나리오

1. 서버는 `/api/images/no-store/:name`에서 항상 `Cache-Control: no-store`를 내려줍니다.
2. 프론트의 Service Worker([client/public/sw.js](client/public/sw.js))가 해당 요청을 가로챕니다.
3. SW는 쿼리스트링을 제거한 경로 기준으로 캐시 키를 정규화해 저장합니다.
4. 이후 `?v=2`, `?v=3`처럼 URL 쿼리가 바뀌어도 동일 경로면 SW 캐시를 반환합니다.
5. `/api/stats`의 `networkHits`가 증가하지 않는 것으로 네트워크 우회 캐시를 확인할 수 있습니다.

## 실행 방법

### 1) 백엔드 실행

```bash
cd server
npm run dev
```

- 서버: `http://localhost:4000`

### 2) 프론트 실행

```bash
cd client
npm run dev
```

- 프론트: `http://localhost:5173`
- Vite dev proxy가 `/api` 요청을 `http://localhost:4000`으로 전달합니다.

## 확인 포인트

1. 페이지 최초 로드 시 이미지 1회 네트워크 요청.
2. `쿼리 버전 증가` 버튼으로 이미지 URL 쿼리를 바꿔도 SW가 캐시된 이미지를 반환.
3. `서버 네트워크 히트 확인` 버튼에서 `networkHits` 값이 계속 유지되면 캐시 재사용 성공.

> 참고: 이 데모는 학습 목적이며, 실제 서비스에서는 개인정보/민감 데이터에 대해 `no-store` 의도를 우회하지 않도록 주의가 필요합니다.

## Study Note

### Web Worker(워커 스레드)

- 메인 스레드가 속한 렌더러 프로세스 내에서 돌아감
- 메인 스레드가 처리하기 복잡한 연산을 보조
- 탭이 닫히면 메인 스레드와 함께 종료됨

### Service Worker

- 등장 배경
  - 서비스 워커가 생기기 전까지 웹은 인터넷이 끊기면 아무것도 할 수 없었다 → 인터넷 연결 여부와 상관없이 항상 작동하는 웹(PWA)을 만들기 위해 등장
- 특징
  - 독립된 실행환경(Worker Context)
    - 웹페이지(메인스레드)와 분리된 별도의 서비스 워커 프로세스라는 곳에서 별도의 생명주기를 가지며 작동 → 백그라운드 실행 가능
  - 가로채기(Network Proxy)
    - 서비스 워커는 브라우저와 네트워크 사이에 존재하는 중간 관리자로서, 백그라운드에서 실행하는 스크립트 파일임
    - 자신이 제어하는 페이지에서 발생하는 이벤트를 수신할 수 있음
    - 브라우저와 네트워크 사이의 요청을 가로챌 수 있음
      - 서버로 갈 요청을 캐시로 되돌리거나 가짜 응답(mock response)를 보내는 등 ‘네트워크 흐름 재설계’가 가능
      - 네트워크에 관한 다양한 권한을 가짐(푸시 알림, 백그라운드 동기화 등)
      - 악성 서비스워커를 심을 수 있기 때문에 반드시 HTTPS 환경에서만 설치 및 작동 가능(localhost 제외)
  - 이벤트 기반의 생명주기(Event-driven & Asynchronous)
    - 서비스워커는 사용하지 않을 땐 브라우저가 종료시켜 메모리 절약
    - fetch나 push와 같이 등록된 이벤트가 트리거되면 다시 실행되어 일을 처리하는 등 ‘필요할 때만’ 깨어남
  - 생명주기
    1. 등록
       - 브라우저에게 서비스 워커 자바스크립트 파일이 어디에 위치해 있는지 알려주는 과정
       - 메인 자바스크립트 파일에서 `navigator.serviceWorker.register(파일경로)` 를 사용
    2. 설치
       - 서비스 워커가 처음 등록되거나 업데이트된 서비스 워커 파일이 감지될 때 발생
       - install 단계 콜백에서 에러가 발생하더라도 서비스 워커는 활성화 단계로 넘어갈 수 있으므로 중요한 작업을 수행하는 경우, waitUntil() 메서드로 설치 단계에서 원하는 작업이 모두 완료된 후에만 활성화되도록 해야 함
    3. 대기 중
       - 서비스 워커 스크립트가 업데이트되어 새로운 버전의 서비스 워커가 설치되었지만, 기존 서비스 워커가 여전히 페이지를 제어하고 있을 때 발생하는 단계
         - 바로 활성화시키지 않는 이유는, 동일한 사이트에서 서로 다른 버전의 서비스 워커가 충돌하는 것을 방지하기 위함
         - 모든 탭(웹페이지)가 닫히거나, 코드로 skipWaiting()을 호출하여 이 단계를 뛰어넘을 수 있다
    4. 활성화
       - 설치가 성공적으로 끝나고, 현재 활성화된 서비스 워커가 없다면 즉시 '활성화' 단계가 됨
       - 활성화 중(Activating), 활성화됨(Activated) 두 단계로 나뉨
         - '활성화중'은 서비스 워커가 본격적으로 활성화되기 직전의 상태
           - install 단계에서와 비슷하게 waitUntil() 메서드를 활용하여 지정된 프로미스 작업이 끝난 뒤에 '활성화됨' 상태로 변경할 수 있음
         - '활성화됨'은 서비스 워커가 활성화된 상태. 이제 서비스 워커는 앱을 제어할 수 있고 fetch 이벤트와 같은 동작 이벤트를 받아서 처리할 수 있다
    5. 중복
       - 서비스 워커가 설치 중 실패하거나 새로운 버전으로 교체되면 중복 상태가 된다. 이 상태의 서비스 워커는 앱에 아무런 영항을 미칠 수 없다.

### 서비스 워커로 리소스 캐싱하기

- fetch() 이벤트를 가로채어 Cached Storage에 HTTP 요청(Request)과 응답(Response) 객체를 쌍으로 저장한 값의 Hit 여부에 따라 캐시된 리소스를 응답으로 반환하거나, 서버로부터 새 리소스를 가져올 수 있다
- 다만 서비스 워커도 브라우저 컨텍스트에서 실행되는 자바스크립트 파일이기 때문에 CORS 보안 정책을 따른다
  - 이 때문에 서버 응답 헤더에 Access-Control-Allow-Origin이 적절히 설정되지 않은 교차 출처 리소스를 요청할 경우 브라우저는 보안을 위해 응답의 세부 내용을 숨기는 'Opaque Response'를 반환하고, 이 응답은 status가 0이기 때문에 캐싱 성공 조건에 status 0을 포함하도록 로직을 수정해야 캐싱이 동작한다
  - 다만 이 방법은 클라이언트 사이드에서 임시방편으로 처리한 거라 리소스의 유효성 면에서 안정적인 캐시 전략이라고 할 수 없다
  - 서비스 워커가 리소스의 유효성을 정확히 판단할 수 있도록 서버 측에서 CORS 헤더를 설정하거나, 커스텀 프록시 서버를 구축하는 방향이 옳다
- 결론적으로 서비스워커로 이미지와 같은 고용량 리소스를 캐싱하는 것은 아주 좋은 방법은 아니지만, 이 아티클에서는 순수 브라우저 리소스만으로 서버에서 캐싱 금지된 리소스를 캐싱해보도록 제약을 둔 데 의의가 있다
